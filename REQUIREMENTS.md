# Rayboard Feature Scope: Jira-Like POC

## Summary

Expand Rayboard from runtime/auth foundation into a Jira-like ticket system with projects, tickets, boards, backlog, sprint entities, roadmaps, and typed custom fields. The app remains one Go binary with `combined`, `frontend`, and `backend` modes, and the frontend talks to the backend API over HTTP in all split-ready paths.

## Key Changes

- Keep the runtime/auth foundation:
  - `rayboard combined`: starts frontend and backend in one process, exposing both UI and API.
  - `rayboard frontend`: starts only the frontend server and talks to the backend over HTTP.
  - `rayboard backend`: starts only the backend HTTP API and SQLite persistence.
  - users are stored in SQLite with username, password hash, disabled flag, and timestamps.
  - startup resets/logs a random POC admin password for the `admin` user and updates the hashed password in SQLite.
  - passwords use a proper password hashing algorithm such as bcrypt or Argon2id.
  - UI authentication uses opaque server-side sessions stored hashed in SQLite and sent to browsers as `HttpOnly` session cookies.
  - API tokens are generated by users, shown once, stored hashed, and accepted as bearer credentials.
  - users with the required RBAC permissions can create, delete, and disable users.
- Add shared authentication and authorization:
  - `POST /api/login` verifies username/password and creates a server-side session for browser UI users.
  - session cookies are `HttpOnly`, `SameSite=Lax`, and `Secure` when served over HTTPS.
  - sessions store user, expiry, created time, last-used time, and revocation state.
  - `POST /api/logout` revokes the current session and clears the cookie.
  - UI and HTMX requests authenticate through the session cookie.
  - API scripts and external clients authenticate with `Authorization: Bearer <token>`.
  - bearer token records store owner, name, created time, last-used time, optional expiry, revoked time, and only hashed token material.
  - disabled users cannot login, use existing sessions, create tokens, use existing tokens, or run owned cron jobs.
  - sessions, bearer tokens, and cron job owners resolve to the same authenticated principal model.
  - every protected operation checks effective RBAC permissions from user role bindings and group role bindings.
  - mutating cookie-authenticated requests require CSRF protection.
  - bearer-token requests do not require CSRF because bearer credentials are not automatically attached by browsers.
  - unauthenticated UI page requests redirect to login; unauthenticated API requests return `401`.
  - authentication and authorization remain backend-owned so split frontend/backend mode does not duplicate password, session, token, or permission logic.
- Use a scalable backend HTTP API layout:
  - avoid large centralized route files as the API surface grows.
  - the top-level backend router only assembles resource route providers and shared infrastructure.
  - resource-focused HTTP packages own route registration and high-level handlers for one API area.
  - route handlers stay thin: decode input, authenticate/authorize, call domain services, and encode output.
  - domain services and repositories own business rules, validation, transactions, persistence, and SQLite access.
  - do not place raw DB CRUD in HTTP route packages.
  - target layout:

```text
internal/backend/httpapi/
  router.go
  shared/
  projects/
    routes.go
    schema.go
    provider.go
  tickets/
    routes.go
    schema.go
    provider.go
  boards/
    routes.go
    schema.go
    provider.go
```

  - `routes.go` contains route registration and high-level HTTP handler methods.
  - `schema.go` contains Huma/OpenAPI-facing request and response DTOs.
  - request DTOs use the `Input` suffix; response DTOs use the `Output` suffix.
  - `provider.go` contains dependency wiring and helpers for that resource package.
  - `/api/projects` and closely project-scoped settings can live under the project route package.
  - top-level resources such as `/api/tickets/{ticket_id}`, `/api/boards/{board_id}`, and `/api/sprints/{sprint_id}` should have their own route packages.
  - avoid deeply recursive package trees unless a subresource becomes large enough to justify its own package.
  - migrate toward this layout incrementally; the existing tracker service can remain during the transition.
- Add proper RBAC with groups:
  - model users, groups, group memberships, roles, permissions, and role bindings in SQLite.
  - role bindings can target either a user or a group.
  - role binding scopes are global and project for the first version.
  - authorization is deny-by-default.
  - replace the global admin flag with a built-in global admin role.
  - bootstrap `admin` user receives the built-in global admin role.
  - built-in roles include global admin, global user manager, project owner, project admin, project member, project viewer, automation manager, and notification manager.
  - built-in roles are seeded/migrated and cannot be deleted; their permission sets can be inspected.
  - permission namespaces include `users:*`, `groups:*`, `roles:*`, `projects:*`, `tickets:*`, `comments:*`, `attachments:*`, `sprints:*`, `boards:*`, `fields:*`, `views:*`, `notifications:*`, `webhooks:*`, `automations:*`, `settings:*`, `demo:*`, and `ai:*`.
  - project ownership is represented as a project-scoped role binding, not as a separate special case.
  - group membership changes immediately affect existing sessions, API tokens, cron jobs, webhook actors, Lua hooks, AI automations, and notification hooks.
  - global admins and global user managers can manage groups and group memberships according to RBAC permissions.
  - project admins can bind allowed users or groups to project roles when they have the project member-management permission.
- Add core work tracking:
  - projects contain tickets, workflow statuses, custom field definitions, sprints, boards, components, versions/releases, roles, members, and board/backlog ordering.
  - projects have owner/lead role bindings.
  - projects support project-scoped role bindings such as owner, admin, member, and viewer.
  - project permissions cover browse, create, edit, manage settings, manage boards, and manage sprints.
  - components group project work by area and may have an owner/default assignee.
  - versions/releases group work for release planning and may have status plus target/release dates.
  - tickets support title, description, status, priority, assignee, reporter, labels, dates, comments, activity history, and optional parent epic.
  - ticket labels are a string array on ticket create/update/list/get/search payloads; labels are normalized to lowercase slugs, deduplicated, and stored as ticket labels.
  - label management UI and separate label CRUD endpoints are not required for the first labels slice.
  - statuses drive board columns; tickets can be reordered in backlog and board views.
- Add durable history and event logging:
  - keep `ticket_activity` as the user-facing timeline for tickets and epics.
  - write timeline entries for visible ticket/epic actions such as create, update, status change, assignment, sprint change, comment add/delete, attachment upload/delete, label change, and automation-visible mutations.
  - add a durable append-only `domain_events` table for backend event/outbox behavior.
  - domain events include type, actor, project, subject type/id, optional related object type/id, JSON payload, occurrence time, and processing metadata.
  - use durable domain events for notifications, webhooks, automation triggers, search/FTS refreshes, integrations, and retryable async processing.
  - the in-memory Go event bus may dispatch events, but it is not the source of truth and must not be the only record of meaningful mutations.
  - keep `audit_log` separate for security/admin-sensitive actions such as login failures, token creation/revocation, user disablement, RBAC changes, settings changes, webhook token rotation, and OpenRouter key changes.
  - do not mix internal-only domain events, user-facing ticket timelines, and security audit entries into one table.
- Add ticket attachments:
  - users with ticket edit/comment permission can attach files to tickets.
  - store attachment metadata in SQLite: ticket, uploader, filename, content type, size, checksum, created time, and deleted time.
  - store attachment bytes in SQLite for the first version to preserve single-file portability.
  - enforce max attachment size and allowed content-type policy from admin settings.
  - include attachment upload/delete events in ticket activity history.
- Add notifications with reusable Shoutrrr destinations:
  - create in-app notifications for assignment, mentions, comments, watched tickets, status changes, sprint changes, release changes, and automation failures.
  - support user and project notification preferences.
  - use `github.com/containrrr/shoutrrr` for external notification delivery.
  - store named Shoutrrr destinations with type, Shoutrrr URL/config, secret fields, enabled flag, owner/scope, timestamps, and last delivery status.
  - store destination secrets securely and redact sensitive fields after creation.
  - support global destinations managed by global admins and reusable across the instance.
  - support project destinations managed by project owners/admins and reusable within that project.
  - support dashboard/view destinations where dashboard/view notifications exist.
  - allow projects to use globally shared destinations plus project destinations.
  - allow dashboards/views to use globally shared destinations, project destinations for their project, and their own local destinations.
  - define notification policies at global, project, and dashboard/view scope.
  - queue external notifications with retry/backoff, delivery history, manual retry, and test-send support.
- Add Lua/AI notification hooks:
  - notification hooks can filter, suppress, transform, enrich, and route notifications before external Shoutrrr delivery.
  - hooks receive event context, candidate recipients, user preferences, project metadata, default message payload, and visible destination names.
  - hooks return a validated notification plan; Shoutrrr only receives validated output.
  - hooks route to named destinations, e.g. `route.to_destination("team-slack")`, and never receive raw Shoutrrr URLs or secrets.
  - notification hooks support the shared nested `engine` object like other automation surfaces.
  - enforce timeout, max script/prompt size, max log size, max recipients, and max generated deliveries.
- Add saved views:
  - users can save personal CEL/FTS filters with sort, columns, grouping, and board/list/backlog display mode.
  - users with project view-management permission can create shared project views.
  - saved views can be pinned to project navigation or used as board filters.
- Add full-text search:
  - use SQLite FTS5 for ticket title, description, comments, and attachment metadata where useful.
  - combine FTS text query with CEL filters and API sort/pagination.
  - expose full-text search in both UI and API.
- Add settings pages:
  - global admin settings cover auth/session policy, attachment limits, notification destinations, webhook allowlists, OpenRouter, demo mode warning, backup/export controls, and system health.
  - project settings cover members, roles, workflow, fields, boards, components, versions, sprints, hooks, webhooks, custom create pages, notifications, saved views, and future CSS override metadata.
  - user settings cover profile, sessions, API tokens, notification preferences, and saved views.
- Add sprint support now:
  - implement sprint CRUD, start/complete states, sprint goal, date range, ticket assignment, and backlog ordering.
  - defer sprint reports such as burndown, velocity, and burnup until later.
- Add roadmap support using `Epic -> tickets`:
  - epics have start/due dates and appear on a timeline.
  - child tickets roll up basic progress by status.
  - dependencies can be deferred unless needed for the first UI.
- Add typed per-project custom fields:
  - users with project field-management permission define fields per project with name, type, required flag, and options where applicable.
  - initial field types: text, number, boolean, date, single-select, multi-select, and user.
  - ticket create/update validates required fields and field value types.
  - store values in a queryable SQLite-friendly structure, not as opaque ticket JSON only.
- Add CEL-based search and filtering:
  - use CEL via `github.com/google/cel-go` as Rayboard's query expression language.
  - parse and type-check CEL expressions, then translate the supported subset into safe parameterized SQLite queries.
  - support comparisons, boolean operators, string helpers, lists, dates, and approved functions such as `currentUser()`, `today()`, and `now()`.
  - expose custom fields through a stable `custom` map/object and validate names/types against project metadata.
  - do not expose direct SQL, arbitrary host access, or an unrestricted CEL function registry.
- Add Lua cron jobs:
  - use `github.com/robfig/cron/v3` for scheduling cron expressions, time zones, job registration, shutdown, and non-overlap handling.
  - use `github.com/yuin/gopher-lua` as the pure-Go Lua VM so Rayboard remains a single Go binary without CGO.
  - store cron jobs in SQLite with owner, name, schedule, timezone, enabled flag, Lua source, last run status, timestamps, and run history.
  - run each job as its owner; disabled users cannot run cron jobs.
  - jobs inherit the owner's current effective RBAC permissions at execution time and cannot exceed what the owner could do through the API.
  - expose only a Rayboard Lua sandbox with functions such as `rayboard.search`, `rayboard.get_ticket`, `rayboard.create_ticket`, `rayboard.update_ticket`, `rayboard.comment`, and `rayboard.log`.
  - do not expose filesystem, shell, raw sockets, arbitrary HTTP, or direct SQLite access to Lua scripts.
  - enforce per-run timeout, max log size, max API response size, and no overlapping runs by default.
- Add shared Lua JSON and Go bridge support:
  - every Lua-capable surface uses one shared sandbox runtime package and one shared Go<->Lua conversion layer.
  - expose a sandboxed JSON module as `json` and `rayboard.json` with `json.encode(value)` and `json.decode(text)`.
  - JSON decode converts objects to Lua tables with string keys, arrays to 1-indexed Lua array tables, booleans/strings/numbers directly, and JSON null to a stable `json.null` sentinel.
  - JSON encode converts Lua array tables to JSON arrays, string-key tables to JSON objects, `json.null` to JSON null, and rejects mixed-key tables, functions, userdata, threads, recursive tables, non-finite numbers, and unsupported values.
  - enforce max JSON input bytes, max encoded output bytes, max nesting depth, and clear validation errors for invalid JSON or unsupported Lua values.
  - Go-backed Rayboard functions exposed to Lua accept plain Lua tables and return plain Lua tables plus errors, e.g. `local ticket, err = rayboard.create_ticket({ title = "Bug" })`.
  - Go service DTOs and API payloads should use the same table/JSON conversion rules instead of one-off reflection or ad-hoc map handling.
  - Lua scripts must never receive raw Go pointers, unrestricted userdata, DB handles, HTTP clients, or Shoutrrr/OpenRouter secrets.
  - table-to-table helper functions should exist for common Rayboard payloads so cron jobs, hooks, custom create pages, webhooks, and notification hooks behave consistently.
  - provide Lua examples for JSON encode/decode, `json.null`, API function calls, validation errors, and safe payload transformation.
- Add Lua ticket hook plugins:
  - use the existing GopherLua stack for project-scoped ticket hooks.
  - hooks are managed by project owners/admins.
  - store hooks in SQLite with project, name, event, phase, enabled flag, Lua source, order, timestamps, and last error.
  - initial hook events cover ticket create and ticket update.
  - hook phases are `before` and `after`.
  - `before` hooks run before persistence and may validate or transform the pending ticket payload.
  - `before` hooks can reject the operation with a user-facing validation message.
  - `after` hooks run after a successful commit and may inspect/log but cannot change committed data.
  - multiple hooks run deterministically by configured order, then name/id.
  - use a stricter hook sandbox than cron jobs: expose only hook context, ticket payload helpers, validation helpers, and logging.
  - do not expose filesystem, shell, raw sockets, arbitrary HTTP, direct SQLite, or the general Rayboard API client to ticket hooks.
  - enforce short timeout, max log size, max script size, and max transformed payload size.
- Add custom ticket create pages:
  - scope create pages to a project and let project owners/admins manage them.
  - authenticated users can submit create pages; tickets are created as the submitting user.
  - submissions use the normal backend ticket create path, including permissions, custom field validation, and Lua `before` ticket hooks.
  - store create pages with project, name, slug, enabled flag, description, optional target ticket type/status, field layout, defaults, timestamps, and owner/admin metadata.
  - field layout supports sections, ordered fields, labels/help text, required overrides, hidden fields, default values, and allowed options.
  - fields can include core ticket fields and project custom fields.
  - optional Lua form logic can compute dynamic form schema, defaults, and options from user/project/context.
  - Lua form logic does not render raw HTML; Rayboard renders all controls from a validated form schema.
  - use a strict form sandbox with context, form helpers, validation helpers, logging, and limited read-only metadata access.
  - do not expose filesystem, shell, raw sockets, arbitrary HTTP, direct SQLite, the general Rayboard API client, or raw HTML injection to custom form Lua.
  - validate submitted fields server-side against the resolved form schema and project field definitions.
  - apply hidden/defaulted fields on the server instead of trusting browser-submitted values.
  - rejected submissions return field-level errors for HTMX partial rendering.
- Add Lua-powered webhooks:
  - support project-scoped incoming and outgoing webhooks managed by project owners/admins.
  - store webhook definitions in SQLite with project, name, direction, enabled flag, owner, Lua source, auth config, last error, timestamps, and delivery/run history.
  - incoming webhooks expose stable generated endpoints such as `POST /api/webhooks/incoming/{id}`.
  - incoming webhooks authenticate with `Authorization: Bearer <webhook-token>`.
  - incoming webhook tokens are shown once on creation/rotation and stored only as hashes.
  - incoming webhook Lua runs after authentication to validate/transform request body, headers, and query params.
  - incoming webhook Lua may create/update tickets only through a constrained Rayboard action API using the webhook's configured actor user and effective RBAC permissions.
  - incoming webhooks reject requests with clear status codes for auth failure, Lua validation failure, malformed payloads, and action failures.
  - outgoing webhooks trigger from selected events such as ticket created, ticket updated, status changed, comment added, sprint changed, and release changed.
  - outgoing webhook Lua builds outbound method, URL path/query, headers, and JSON body from event context.
  - outgoing webhooks send through a controlled HTTP client with configured destination allowlist, timeout, max payload size, and no arbitrary network access beyond allowed targets.
  - outgoing webhooks support queued delivery, retry with backoff, delivery history, and manual redelivery.
  - outgoing webhooks may optionally sign requests with a stored secret or static header token.
  - use a webhook-specific GopherLua sandbox with request/event context, JSON helpers, validation helpers, controlled Rayboard actions for incoming, outbound request builder for outgoing, and logging.
  - do not expose filesystem, shell, raw sockets, direct SQLite, unrestricted HTTP, or the full Rayboard API client to webhook Lua.
  - enforce timeout, max script size, max log size, max request/response body size, and max generated action count.
- Add OpenRouter AI automation alongside Lua:
  - everywhere Rayboard supports Lua automation, also support AI automation with the same reusable engine object.
  - automation definitions use `engine.type` as the discriminator with `lua` and `ai` values.
  - Lua engine definitions use `engine.script`.
  - AI engine definitions use `engine.prompt` and `engine.provider_id`.
  - `engine.provider_id` references an admin-managed OpenRouter provider configuration containing model, API key/secret material, limits, and defaults.
  - reuse this engine object for cron jobs, ticket hooks, custom create pages, incoming webhooks, outgoing webhooks, and notification hooks.
  - OpenRouter is the only supported AI provider in the first version.
  - global admins configure the OpenRouter API key, default model, allowed models, default timeout, and usage limits.
  - store the OpenRouter API key securely and never expose it to project users.
  - project owners/admins may select only allowed OpenRouter models for AI automations.
  - AI automations store prompt, optional system instructions, selected OpenRouter model, output schema, timeout, and last error.
  - AI responses must be JSON matching the declared schema for the automation surface.
  - validate AI output before applying transformations, actions, outbound requests, or form schemas.
  - invalid JSON/schema failures are treated as runtime errors and recorded in history.
  - AI cron jobs receive job context and may return allowed Rayboard actions.
  - AI ticket hooks receive old/proposed ticket context and may return validation errors or transformed payload for `before` hooks; `after` hooks may only return logs/observations.
  - AI custom create pages receive user/project/context and return form schema/defaults/options, never raw HTML.
  - AI incoming webhooks receive authenticated request data and return validation result plus allowed Rayboard actions.
  - AI outgoing webhooks receive event context and return outbound request shape.
  - AI automations use the same owner/actor, RBAC permission, timeout, log, payload-size, and action-count limits as the equivalent Lua surface.
  - AI output never bypasses normal backend validation, custom field validation, ticket hooks, or authorization.
  - record prompt input summary, OpenRouter model, usage metadata when available, validated output, errors, and resulting actions in run history.
  - provide test/preview execution for every AI automation before enabling.
- Use a lightweight embedded frontend:
  - build the UI with server-rendered Go `html/template` views plus HTMX-enhanced interactions.
  - use plain CSS and small vanilla JavaScript modules for local behavior.
  - use SortableJS as a vendored static asset for board/backlog drag-and-drop.
  - use CodeMirror as an optional vendored static asset for Lua and CEL editors; a plain `<textarea>` is acceptable for the first pass if needed.
  - embed templates and static assets into the Go binary using Go's `embed.FS`.
  - avoid a Node/npm build pipeline for the first version.
- Defer project/board CSS customization:
  - do not implement a project-owner CSS editor, board-owner CSS editor, custom CSS upload, or raw stylesheet storage in the first version.
  - default embedded CSS remains the only active stylesheet system in the first version.
  - preserve a future CSS override extension point with stable project/board wrapper classes, useful data attributes, and predictable CSS variable names where practical.
  - avoid hard-coding inline styles except for unavoidable dynamic state such as status colors.
- Add a demo seed command:
  - `rayboard demo seed --backend-url http://127.0.0.1:8081 --admin-user admin --admin-password <password> --fresh-reset`
  - seeds a running backend over HTTP rather than writing directly to SQLite.
  - logs in as admin, then calls normal backend API endpoints so validation, permissions, hooks, custom fields, and activity logging are exercised.
  - supports destructive fresh reseeding only when `--fresh-reset` is explicitly present.
  - refuses destructive reset without `--fresh-reset`.
  - reset/seed operations require global admin.
  - creates multiple demo users with random passwords, groups, global role bindings, and project role bindings.
  - logs generated demo credentials to stdout/stderr in a clear demo-only table after seeding.
  - marks seeded objects with a demo marker where useful so future cleanup can identify them.
  - seeds groups such as Product, Engineering, QA, Viewers, Automation, and Managers.
  - seeds projects, owners, roles, role bindings, components, versions/releases, workflows/statuses, boards, backlog ordering, sprints, epics, tickets, comments, labels, activity history, custom fields, saved CEL query examples, Lua cron jobs, Lua ticket hooks, and custom ticket create pages.
  - prints seeded project/board URLs and summary counts after completion.

## Public Interfaces

- CLI:
  - `rayboard combined [flags]`
  - `rayboard frontend --backend-url http://host:port [flags]`
  - `rayboard backend --db path/to.sqlite [flags]`
  - `rayboard demo seed --backend-url http://host:port --admin-user admin --admin-password <password> --fresh-reset`
- Shared config flags/env:
  - `--frontend-addr`, default `127.0.0.1:8080`
  - `--backend-addr`, default `127.0.0.1:8081`
  - `--backend-url`, default `http://127.0.0.1:8081`
  - `--db`, default `rayboard.sqlite`

  - Redoc UI served under `/api/docs/redoc` with all Redoc static assets embedded into the Go binary; no CDN, Node build, generated runtime file, or external docs server is required.
  - project CRUD
  - project role/member endpoints

  - sprint CRUD, start, complete, and ticket assignment endpoints

## Assumptions

- Sprint support means sprint entities and workflow now, but not advanced agile reporting yet.
- Roadmaps initially mean epic timelines with child ticket rollups, not portfolio-level initiatives.
- Custom fields are typed and project-scoped from the start.
- Kanban board behavior remains supported alongside backlog and sprint planning.
- CEL is used for search/filter expressions, not as a general scripting language.
- Sorting and pagination remain API parameters rather than CEL syntax.
- Automation rules may reuse CEL later, but ticket search is the first integration.
- Cron job creation/management requires the appropriate `automations:*` RBAC permission.
- Capabilities from earlier planning are represented as RBAC permissions granted through user/group role bindings.
- Cron jobs use the internal backend service/client, but only through the same authorization checks as public API calls.
- Arbitrary external HTTP from Lua is out of scope for the first version.
- First plugin system is limited to ticket hooks.
- Ticket hooks are project-scoped and managed by project owners/admins.
- Ticket hooks do not get the full Rayboard API client in the first version.
- Cron jobs remain the mechanism for scheduled cross-resource automation with API access.
- Custom create pages are authenticated-only in the first version.
- Custom create page Lua may compute form schema behavior but does not render HTML.
- Custom create pages reuse the normal ticket create backend path and do not bypass permissions, custom field validation, or ticket hooks.
- Public unauthenticated intake forms are out of scope for the first version.
- Webhooks are project-scoped in the first version.
- Incoming webhooks authenticate with bearer tokens, not HMAC signatures.
- Incoming webhooks act as a configured actor user and cannot exceed that actor's effective RBAC permissions.
- Outgoing webhooks use queued delivery and do not block the original ticket/project operation.
- Arbitrary outbound HTTP is not allowed; outgoing targets must match configured allowlists.
- Store attachment bytes in SQLite for v1; external filesystem/object storage can be added later.
- Use Shoutrrr as the Go-native multi-channel notification library because it fits the single-binary requirement.
- Shoutrrr destinations are reusable named resources.
- Secrets are stored once per destination and never copied into hooks or policies.
- Destination inheritance is global -> project -> dashboard/view.
- Notification hooks route by destination name, not by raw Shoutrrr URL.
- Lua/AI notification hooks produce a validated notification plan and do not call Shoutrrr directly.
- External notifications are queued and do not block ticket operations.
- Saved views store query/display configuration, not arbitrary SQL.
- FTS text query and CEL filters remain separate inputs that are combined by the backend.
- OpenRouter is the only supported AI provider in the first version.
- AI is an alternative engine to Lua, not a replacement.
- AI automations must return structured JSON; free-form text is never applied directly.
- AI output is always validated and authorized by Rayboard before effects occur.
- Demo seeding should not require OpenRouter to be configured.
- Demo seeding targets a running backend over HTTP.
- Demo fresh reset is intentionally destructive and must be explicitly requested.
- Demo command is for local/demo environments, not production use.
- Generated demo user passwords are printed once in the command output.
- Password login is only for human UI sessions.
- API tokens are the supported auth method for scripts and external clients.
- Sessions and API tokens are opaque random secrets, not JWTs.
- Backend remains the source of truth for authentication and authorization.
- The frontend is server-rendered Go templates with HTMX, not a SPA framework.
- Static assets are embedded into the Go binary; no runtime filesystem assets are required.
- No Node/npm build step is required for the first version.
- CodeMirror can be deferred in favor of textareas if vendoring it would slow down the first implementation.
- CSS customization is explicitly deferred.
- Future customization will be implemented as a CSS override layer, scoped to project/board wrappers.
- The first version should preserve enough stable markup structure to make that future override layer practical.
- Combined mode exposes both UI and API; the exact same backend API should remain usable by scripts.
- Combined mode may still use HTTP between frontend and backend internally to preserve the future split boundary.
- Frontend mode requires `--backend-url` or uses its default HTTP URL; it will not create an in-process backend fallback.
- Deleting users can hard-delete only while no ticket ownership/history constraints exist; once tickets exist, disabling users should be preferred and historical references preserved.
- The startup admin password reset is POC-only and must be replaced with durable admin setup before production use.
